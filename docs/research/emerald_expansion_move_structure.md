# pokeemerald-expansion (RHH) — Move Structure Reference

Comparative reference for migrating moves from pokeemerald-expansion to your FireRed ROM hack. Covers structural differences and the mapping between the two codebases.

Local clone: `/mnt/data/Github/rh-hideout/pokeemerald-expansion`

---

## Architecture Comparison

| Aspect | FireRed Decomp | pokeemerald-expansion (RHH) |
|--------|---------------|----------------------------|
| **Move count** | 354 (Gen 1-3 only) | 848 base + Z-Moves + Max Moves (Gen 1-9) |
| **Move stats** | Separate `gMovesInfo[]` array in `battle_moves.h` | Unified `gMovesInfo[]` in `moves_info.h` — **name, description, stats, flags, contest data, and animation pointer all in one struct** |
| **Move names** | Separate file: `move_names.h`, ALLCAPS, max 12 chars | Inline `.name = COMPOUND_STRING("Mixed Case")` in `moves_info.h`. Mixed case. `HANDLE_EXPANDED_MOVE_NAME()` for names >12 chars |
| **Move descriptions** | Separate file: `move_descriptions.c` (string + pointer array) | Inline `.description = COMPOUND_STRING(...)` in `moves_info.h` |
| **Flags** | Bitfield `.flags` (`FLAG_MAKES_CONTACT \| FLAG_PROTECT_AFFECTED \| ...`) | Individual booleans: `.makesContact = TRUE`, `.punchingMove = TRUE`, `.windMove = TRUE`, `.slicingMove = TRUE`, etc. |
| **Category** | `DAMAGE_CATEGORY_PHYSICAL / DAMAGE_CATEGORY_SPECIAL / DAMAGE_CATEGORY_STATUS` | `DAMAGE_CATEGORY_PHYSICAL / DAMAGE_CATEGORY_SPECIAL / DAMAGE_CATEGORY_STATUS` |
| **Secondary effects** | `.effect` + `.secondaryEffectChance` | `.additionalEffects = ADDITIONAL_EFFECTS({.moveEffect = ..., .chance = ...})` macro — supports multiple effects |
| **Animation** | Separate `gBattleAnims_Moves` table in `battle_anim_scripts.s` | Inline `.battleAnimScript = Move_POUND` in `moves_info.h` |
| **Contest data** | Separate `contest_moves.h` | Inline `.contestEffect`, `.contestCategory`, `.contestComboStarterId`, `.contestComboMoves` in `moves_info.h` |
| **AI** | Assembly scripts in `data/battle_ai_scripts.s` (effect-based) | C functions in `src/battle_ai_main.c` (effect-based, same pattern) |
| **Level-up learnsets** | Single file with `LEVEL_UP_MOVE(lvl, move)` macro (9-bit encoding, max move ID 511) | Split into 9 files by gen (`level_up_learnsets/gen_1.h` through `gen_9.h`). Uses `struct LevelUpMove` — **no 9-bit move ID limit** |
| **TM/HM compat** | 64-bit bitmask per species in `tmhm_learnsets.h` | Flat `u16` move arrays per species in `teachable_learnsets.h` — **no bitmask limit**. Auto-generated by `tools/learnset_helpers/teachable.py` |
| **TM/HM list** | `sTMHMMoves[]` in `party_menu.h` | `FOREACH_TM(F)` / `FOREACH_HM(F)` macros in `include/constants/tms_hms.h` |
| **Move IDs** | Simple `#define MOVE_XYZ N` | Same, plus `MOVES_COUNT_GEN1` through `MOVES_COUNT_GEN9` milestones, pre-Gen aliases (`MOVE_FAINT_ATTACK` → `MOVE_FEINT_ATTACK`) |
| **Gen-configurable** | No | `B_UPDATED_MOVE_DATA >= GEN_X` for per-gen stat/PP variations |
| **Z-Move/Dynamax** | No | Full support: `.zMove = { .effect = Z_EFFECT_* }`, separate Z/Max move IDs after `MOVES_COUNT` |
| **Family ifdefs** | No | `#if P_FAMILY_BULBASAUR` wraps all per-family data for conditional compilation |

## Key Files in pokeemerald-expansion

| Purpose | Path |
|---------|------|
| Move IDs | `include/constants/moves.h` |
| Move stats/name/desc/flags/anim (ALL-IN-ONE) | `src/data/moves_info.h` |
| Move effects | `include/constants/battle_move_effects.h` |
| Level-up learnsets | `src/data/pokemon/level_up_learnsets/gen_1.h` through `gen_9.h` |
| Egg moves | `src/data/pokemon/egg_moves.h` |
| TM/HM move list | `include/constants/tms_hms.h` |
| TM/HM/tutor compatibility | `src/data/pokemon/teachable_learnsets.h` (auto-generated) |
| AI logic | `src/battle_ai_main.c` |
| Contest data | Inline in `moves_info.h` |
| Battle animations | Referenced inline via `.battleAnimScript` in `moves_info.h`; scripts in `data/battle_anim_scripts.s` |

## MoveInfo Struct Example (Expansion)

```c
[MOVE_FIRE_PUNCH] =
{
    .name = COMPOUND_STRING("Fire Punch"),
    .description = COMPOUND_STRING(
        "A fiery punch that may burn\n"
        "the foe."),
    .effect = EFFECT_HIT,
    .power = 75,
    .type = TYPE_FIRE,
    .accuracy = 100,
    .pp = 15,
    .target = MOVE_TARGET_SELECTED,
    .priority = 0,
    .category = DAMAGE_CATEGORY_PHYSICAL,
    .makesContact = TRUE,
    .punchingMove = TRUE,
    .additionalEffects = ADDITIONAL_EFFECTS({
        .moveEffect = MOVE_EFFECT_BURN,
        .chance = 10,
    }),
    .contestEffect = CONTEST_EFFECT_HIGHLY_APPEALING,
    .contestCategory = CONTEST_CATEGORY_BEAUTY,
    .contestComboStarterId = COMBO_STARTER_FIRE_PUNCH,
    .contestComboMoves = {COMBO_STARTER_ICE_PUNCH, ...},
    .battleAnimScript = Move_FIRE_PUNCH,
},
```

vs FireRed equivalent (spread across 4+ files):

```c
// battle_moves.h
[MOVE_FIRE_PUNCH] = {
    .effect = EFFECT_BURN_HIT,
    .power = 75,
    .type = TYPE_FIRE,
    .accuracy = 100,
    .pp = 15,
    .secondaryEffectChance = 10,
    .target = MOVE_TARGET_SELECTED,
    .priority = 0,
    .flags = FLAG_MAKES_CONTACT | FLAG_PROTECT_AFFECTED | ...,
    .category = DAMAGE_CATEGORY_PHYSICAL,
},
// move_names.h:        [MOVE_FIRE_PUNCH] = _("FIRE PUNCH"),
// move_descriptions.c: gMoveDescription_FirePunch[] = _("...");
// battle_anim_scripts.s: .4byte Move_FIRE_PUNCH
```

## Migration Guide: Expansion → FireRed

When porting a move from the expansion to your FireRed hack:

### 1. Copy the Move ID
Both use the same numbering for Gen 1-3 moves (1-354). For Gen 4+ moves, assign the next available ID in your `moves.h`.

### 2. Convert MoveInfo → BattleMove
Map the expansion's unified struct to your 4 separate files:

| Expansion Field | FireRed Target |
|----------------|----------------|
| `.name` | `move_names.h` — convert to ALLCAPS, truncate to 12 chars |
| `.description` | `move_descriptions.c` — create string + add pointer |
| `.effect` | `battle_moves.h` `.effect` — most effect names are identical |
| `.power, .type, .accuracy, .pp` | `battle_moves.h` — direct copy |
| `.target, .priority` | `battle_moves.h` — direct copy |
| `.category` | `battle_moves.h` — map `DAMAGE_CATEGORY_*` → `SPLIT_*` |
| `.makesContact` etc. | `battle_moves.h` `.flags` — map booleans back to bitfield |
| `.additionalEffects` | `battle_moves.h` `.effect` + `.secondaryEffectChance` |
| `.battleAnimScript` | `battle_anim_scripts.s` — reuse or create animation |
| `.contestEffect`, `.contestCategory` | Not applicable (FireRed has no contests) |
| `.zMove`, Dynamax fields | Not applicable |

### 3. Flag Mapping
| Expansion Boolean | FireRed Flag |
|------------------|-------------|
| `.makesContact = TRUE` | `FLAG_MAKES_CONTACT` |
| `.protectAffected = TRUE` (default) | `FLAG_PROTECT_AFFECTED` |
| `.mirrorMoveBanned = FALSE` (default) | `FLAG_MIRROR_MOVE_AFFECTED` |
| `.magicCoatAffected = TRUE` | `FLAG_MAGIC_COAT_AFFECTED` |
| `.snatchAffected = TRUE` | `FLAG_SNATCH_AFFECTED` |
| `.punchingMove`, `.windMove`, `.slicingMove`, `.bitingMove`, etc. | No equivalent in vanilla FireRed (custom flag needed) |

### 4. Effect Mapping
Most `EFFECT_*` constants are shared. Key differences:

- Expansion uses `additionalEffects` macro for secondary effects (burn/flinch/stat drops on damage). FireRed encodes these as the primary `.effect` (e.g. `EFFECT_BURN_HIT`, `EFFECT_FLINCH_HIT`).
- Expansion has many more effects for Gen 4-9 moves that don't exist in FireRed's `battle_move_effects.h`. These require implementing new battle scripts.

### 5. Learnset Differences

| System | Expansion Format | FireRed Format |
|--------|-----------------|----------------|
| Level-up | `struct LevelUpMove` in per-gen files | `LEVEL_UP_MOVE(lvl, move)` macro in single file, 9-bit move ID limit |
| TM/HM compat | Flat `u16[]` per species | 64-bit bitmask `TMHM()` per species |
| Teachable | `teachable_learnsets.h` (auto-generated) | Manual `sTMHMMoves[]` + `tmhm_learnsets.h` |

## Constraints When Migrating

| Constraint | Detail |
|------------|--------|
| Move ID cap | FireRed: max 511 (9-bit learnset encoding). Expansion: no hard limit |
| Name length | FireRed: 12 chars ALLCAPS. Expansion: mixed case, supports longer names |
| TM slots | FireRed: 64 max (bitmask). Expansion: unlimited (flat array) |
| New effects | Must implement battle scripts in FireRed's assembly-based system vs expansion's C-based system |
| Gen configs | Expansion's `B_UPDATED_MOVE_DATA >= GEN_X` conditionals must be resolved to a single value for FireRed |
| Animations | Expansion has animations for Gen 4-9 moves that don't exist in FireRed |
| Contest data | FireRed has no contests — skip these fields |

## Checklist: Migrating a Single Move

- [ ] Add `#define MOVE_XYZ` in FireRed `include/constants/moves.h`
- [ ] Extract stats from expansion `moves_info.h` → FireRed `battle_moves.h`
- [ ] Convert `.name` to ALLCAPS ≤12 chars → `move_names.h`
- [ ] Copy/rewrite `.description` → `move_descriptions.c`
- [ ] Check if `.effect` exists in FireRed's `battle_move_effects.h`; if not, implement it
- [ ] Map boolean flags → FireRed bitfield `.flags`
- [ ] Resolve any `B_UPDATED_MOVE_DATA` conditionals to a fixed gen
- [ ] Add animation pointer in `battle_anim_scripts.s` (reuse similar move initially)
- [ ] Update learnsets: level-up, egg moves, TM/HM compat as needed
- [ ] Verify AI handles the effect (no change needed if reusing existing effect)
